# 2-2. 출품작 소개

> 본 절은 사업계획서 「2. 제안배경 및 출품작 소개」의 하위 항목으로,
> 2-1(제안 배경)에서 제기된 ① 장애학생의 고등교육 접근 격차, ② 학내 지원체계의 실효성 한계, ③ 졸업 이후 노동시장 진입 격차 문제에 대한 해결책으로서 본 출품작을 제안한다.

## 1. 출품작 개요

### 1.1 서비스 정의

| 항목 | 내용 |
| --- | --- |
| **서비스명** | **베프 (BeFree)** — *Barrier-Free Friend Campus*. ‘가장 친한 친구(베프)’ 라는 일상어와 ‘배리어프리(Barrier-Free)’ 의 중의적 의미를 결합 |
| **한 줄 정의** | 장애학생의 캠퍼스 내 **이동·관계·정보·안전** 문제를 단일 모바일 플랫폼에서 통합 해결하는 **AI·공공데이터 기반 배리어프리 대학생활 플랫폼** |
| **핵심 사용자** | ① 대학 재학 장애학생(주 사용자) ② 비장애학생(상호지원 참여자) ③ 학내 장애학생지원센터·보호자(지원·안전 주체) |
| **운영 형태** | 모바일 웹/PWA 기반(반응형), 향후 네이티브 앱 확장 |
| **핵심 기술 스택(현 구현)** | React 18 · TypeScript · MUI · React Router · Vite |
| **핵심 차별점** | 장애학생만의 ‘복지 앱’이 아니라, 비장애학생과 학내 지원체계까지 **참여 구조 안으로 통합**한 캠퍼스 공동체 인프라 |

### 1.2 한 문장 요약(엘리베이터 피치)

> **“베프”는 흩어져 있는 캠퍼스의 접근성 정보·지원 인력·동료 학생을 장애학생의 5분 뒤 동선 위에 끌어다 놓는, 모두를 위한 캠퍼스 생활 인프라이다.**

## 2. 문제 정의 — 기존 방식의 한계

2-1에서 살핀 바와 같이 법·제도(장애인차별금지법 제13~15조, 특수교육법 제29~32조)와 인프라(장애학생지원센터, 개인별 교육지원계획)는 일정 수준 마련되어 있다. 그러나 학생의 **하루 단위 생활 문제 해결**에는 도달하지 못한다. 주요 한계는 다음 네 가지로 요약된다.

### 2.1 정보의 파편화

시설 접근성·활동 공고·교내 공지·외부 활동 정보가 각기 다른 채널에 분산되어 있어, 장애학생이 자신에게 필요한 정보를 모으는 데 비장애학생보다 훨씬 큰 탐색 비용을 부담한다. 특히 “이 활동에 휠체어로 참여 가능한가”, “이 강의실까지 엘리베이터로 갈 수 있는가” 같은 **접근성 메타데이터** 는 어디에도 정리되어 있지 않다.

### 2.2 지원 인력의 절대 부족

2025년 대학알리미 공시 기준 장애학생지원센터 인력의 **77.2%가 겸직**, 전담 인력은 22.8%에 불과하다. 이 인력 구조로는 학생의 일상적·실시간 도움 요청을 감당할 수 없으며, 결과적으로 지원 체계는 ‘공식 절차’ 중심으로만 작동한다.

### 2.3 상호지원 구조의 부재

비장애학생이 ‘함께할 의향’을 가져도 참여할 통로가 없고, 신뢰 검증·인센티브 체계도 미비하다. 자발적 봉사에 의존하는 비공식 도움은 지속 가능하지 않다.

### 2.4 긴급 상황 대응 공백

캠퍼스 내 위치 기반의 즉각적 도움 요청 채널이 부재하여, 사고·낙오·갑작스러운 신체 상태 변화 시 도움이 도달하기까지 시간이 지연된다.

### 2.5 접근성을 전제하지 않은 기존 UI

기존 대학 앱·커뮤니티는 비장애학생 중심으로 설계되어 스크린리더·고대비·자막·키보드 내비게이션 등 **WCAG 2.1 AA 수준의 기본 접근성조차 보장되지 않는 경우가 많다.**

> **요약**: 문제는 ‘자원이 없다’가 아니라 **‘자원이 학생에게 도달하지 않는다’**. 본 출품작은 이 도달 경로 자체를 재설계한다.

## 3. 해결 전략 — 네 개의 레이어

‘베프’는 단일 기능 앱이 아니라, 장애학생의 대학생활 전 과정을 관통하는 **네 가지 레이어**로 구성된다. 각 레이어는 단일 앱 내에서 상호 참조되어, 학생이 “정보 탐색 → 사람 연결 → 이동 → 안전 확보”의 동선을 끊김 없이 수행하도록 한다.

```
 ┌──────────────────────────────────────────────────────────────────┐
 │           [4] 안전 레이어   전역 SOS / 위치 자동 공유           │
 ├──────────────────────────────────────────────────────────────────┤
 │  [1] 정보 레이어    [2] 이동 레이어    [3] 관계 레이어            │
 │    HomePage           MapPage             MatchingPage           │
 │   (활동·공지·       (배리어프리         (양방향 동행            │
 │    인기글 큐레이션)   캠퍼스 지도)        매칭 + 신뢰 배지)       │
 ├──────────────────────────────────────────────────────────────────┤
 │              개인 접근성 프로필 / 앱 내장 접근성 설정             │
 │                          ProfilePage                              │
 └──────────────────────────────────────────────────────────────────┘
```

| 레이어 | 해결 대상 | 구현 화면 |
| --- | --- | --- |
| **① 정보 레이어** | 학내 공지·대외활동·커뮤니티 인기글을 ‘접근 편의 정보’ 메타데이터와 함께 단일 화면에 큐레이션 | `HomePage` |
| **② 이동 레이어** | 캠퍼스 지도 위 건물·엘리베이터·경사로·접근가능 화장실을 카테고리 필터·상세 정보·길찾기로 제공 | `MapPage` |
| **③ 관계 레이어** | 장애학생(이용자) ↔ 비장애학생(제공자)의 양방향 동행 매칭, 인증·매너 점수·인센티브로 신뢰 담보 | `MatchingPage` |
| **④ 안전 레이어** | 전 화면 어디서나 호출 가능한 SOS — 보호자·지원센터에 현재 위치 자동 통보 | `SosDialog`(전역) |
| **공통** | 개인 접근성 프로필 등록 + 앱 내장 접근성 설정(고대비·모션·자막·스크린리더) | `ProfilePage` / `AccessibilityDialog` |

## 5. 사용자 시나리오

### 시나리오 A. 휠체어 이용자 ‘김OO’ 의 하루

1. 아침 — **HomePage** 에서 ‘배리어프리 캠퍼스 서포터즈 2기’ 모집을 ‘엘리베이터·수어통역’ 접근 태그와 함께 확인.
2. 1교시 이동 — **MapPage** 에서 인문관까지 ‘엘리베이터·경사로’ 필터로 경로 확인 후 길찾기 실행.
3. 점심 — **MatchingPage** 에 ‘점심시간 식당 동행’ 요청 게시. 학교 이메일 인증·매너 4.8점의 ‘이OO(제공자)’ 와 매칭.
4. 오후 — 중앙도서관 3층에서 갑작스러운 신체 이상. 우측 하단 **SOS 버튼** 1회 탭 → 보호자·장애학생지원센터에 위치 자동 통보.
5. 저녁 — **ProfilePage** 에서 이번 주 누적 매칭·봉사시간 인정 내역 확인.

### 시나리오 B. 비장애학생 ‘이OO’ 의 참여

1. **MatchingPage** ‘제공자’ 탭에서 본인이 가능한 시간대의 요청 확인.
2. ‘동행 제공하기’ 신청 → 매칭 확정.
3. 완료 후 **봉사시간 자동 인정**, 학교 공동 인증 활동이면 **학점 연계** 배지 부여.
4. **ProfilePage** 의 ‘제공 가능 역량(수어·필기 도우미 등)’ 항목이 누적되며 신뢰도 상승.

## 6. AI · 공공데이터 활용 방안

본 출품작은 신규 데이터를 ‘새로 만든다’기보다 **이미 존재하는 공공·교내 데이터를 학생 일상의 의사결정 지점에 ‘배달’** 하는 방향으로 AI·공공데이터를 활용한다.

### 6.1 활용 공공데이터

| 데이터 | 출처 | 활용 영역 |
| --- | --- | --- |
| 장애인 편의시설 정보 | 보건복지부 / 한국지능정보사회진흥원 | 캠퍼스·주변 시설 마커 및 접근 편의 상세 |
| 저상버스·교통약자 이동지원 데이터 | 국토교통부 | 통학·이동 경로 안내 |
| 등록장애인 현황 통계 | 보건복지부(국가승인통계 117061) | 대학별 수요 분석, 지원 우선순위 산정 |
| 장애대학생 교육복지지원 실태조사 | 교육부 / 국립특수교육원 | 대학별 지원 수준 가시화(중장기) |
| 학사 공지·시설 공시 | 대학알리미 / 각 대학 | ‘교내 소식’ 자동 큐레이션 |

### 6.2 AI 활용

1. **개인화 매칭·추천** — 사용자 접근성 프로필(필요 도움·제공 역량)과 활동·동행 게시글의 접근성 태그를 매칭하여 최적 항목 추천.
2. **접근성 정보 신뢰도 보정** — 사용자 제보·후기를 분석해 시설 접근성 정보의 정확도·최신성을 자동 보정(예: “엘리베이터 점검 중” 키워드 자동 감지).
3. **SOS 우선 배정** — 위치·시간·과거 패턴을 기반으로 가장 빠르게 도달 가능한 지원 주체를 자동 선정.
4. **비정형 텍스트 분석** — 학내 공지·커뮤니티 글에서 ‘이동 동선 변경’, ‘엘리베이터 점검’ 등 **접근성 영향 키워드 자동 추출 → 영향받는 학생에게 알림**.

> AI는 “장애학생 전용 별도 데이터를 새로 수집”하는 데 쓰이지 않는다. **이미 있는 데이터를 장애학생 관점에서 재해석·재배치**하는 도구로 활용된다.

## 7. 접근성 설계 원칙 (Accessibility by Design)

본 출품작은 ‘접근성 기능을 추가한 앱’이 아니라 **처음부터 접근 가능한 UI로 설계된 앱**이다. 모든 화면에 다음 원칙을 코드 레벨에서 일관되게 적용했다.

| 원칙 | 구현 내용 |
| --- | --- |
| **인지 가능(Perceivable)** | 모든 상호작용 요소에 `aria-label` 부여, 장식 요소는 `aria-hidden`, 색상 대비 4.5:1 이상 |
| **운용 가능(Operable)** | 키보드 포커스 outline 명시(`:focus-visible`), 최소 터치 영역 44~48px, 드래그 가능 영역은 키보드 대체 조작 제공 |
| **이해 가능(Understandable)** | 모든 영역 `lang="ko"`, 동적 콘텐츠는 `aria-live`로 안내, 상태 변화는 라벨 텍스트로도 전달 |
| **견고함(Robust)** | 시맨틱 HTML(`<main>`, `<section>`, `<header>`, `role="list"` 등) 일관 적용 |
| **사용자 제어** | 고대비·모션 줄이기·스크린리더 최적화·자막 기본 표시를 **앱 내장 설정**으로 제공 |

이로써 본 앱은 그 자체로 **‘배리어프리 모바일 디자인의 레퍼런스’** 가 될 수 있다.

### 8.2 질적 차별성

1. **‘장애학생 전용 복지 앱’이 아닌, 캠퍼스 공동체 플랫폼**
   기존 서비스가 ‘지원 대상–제공 기관’의 이원 구조였다면, ‘베프’는 **장애학생–비장애학생–학내 지원체계의 삼각 구조**를 형성한다. 봉사시간·학점 인센티브와 학교 공동 인증 배지가 비장애학생의 자발적 참여를 제도화한다.

2. **제도가 보장한 자원을 ‘학생 손 안’ 으로 옮긴다**
   장애인차별금지법·특수교육법이 보장하는 권리와 장애학생지원센터의 자원은 이미 존재한다. 본 출품작은 그 자원이 **학생의 5분 뒤 동선**에서 실제로 작동하도록 만든다.

3. **처음부터 접근 가능한 UI**
   ARIA·키보드·터치 영역·시각 숨김 텍스트가 모든 화면에 일관 적용되어, 앱 자체가 접근성 레퍼런스가 된다.

4. **데이터 자산화 가능성**
   사용자가 시설을 이용하고 동행을 매칭할수록 **접근성 데이터·매칭 패턴·지원 수요 데이터**가 누적된다. 이는 학내 지원센터의 인력 배치, 시설 개선 우선순위, 정부 실태조사 보완에 활용될 수 있는 **장애 친화 캠퍼스 데이터 자산**으로 발전한다.

5. **확장성**
   대학 캠퍼스 → 공공기관·박물관·대형 의료기관 등 ‘넓고 복잡한 공간을 이용하는 장애인’ 전 영역으로 동일 구조 확장이 가능하다.
